PHÂN TÍCH KIẾN TRÚC & CODEBASE

1.1. Tổng quan Hệ thống

Thông tin Chi tiết
Nền tảng Flutter (Dart SDK ^3.10.4)
Database SQLite local (sqflite ^2.4.2)
State Management ChangeNotifier (MVVM thủ công)
Email Service SMTP trực tiếp qua package mailer
Hashing SHA-256 (package crypto)
DB Version 7 (với migration incremental)

1.3. Design Patterns đang áp dụng

Pattern Vị trí triển khai Đánh giá
Repository Pattern data/interfaces/ → data/implementations/ ✅ Tách abstract khỏi implementation, dễ test
Singleton DatabaseHelper.instance, EmailService._instance, AuthViewModel.instance ✅ Đảm bảo duy nhất 1 instance cho DB và services
MVVM Views → ViewModels → Repositories ✅ Separation of concerns rõ ràng
DTO / Data Mapper dtos/ (Model extends Entity, fromJson/toJson) ✅ Mapping DB ↔ Domain sạch
Observer ChangeNotifier + notifyListeners() ✅ Reactive UI binding chuẩn Flutter
Factory Method AuthViewModel() factory constructor ✅ Đảm bảo single instance
Template Method DatabaseHelper._onUpgrade() migration chain ✅ Migration theo version rõ ràng
Soft Delete MedicalDocumentRepositoryImpl (is_deleted, status_before_soft_delete) ✅ Thiết kế an toàn dữ liệu tốt
Audit Trail Ghi audit_logs trong mọi thao tác CRUD quan trọng ✅ Truy vết hành vi người dùng
Self-Healing PatientRepositoryImpl.getFamilyMembers() tự sửa family_id null ✅ Tự động khắc phục data inconsistency

PHÂN TÍCH LUỒNG (FLOW ANALYSIS)

2.1. Luồng Chính: Tạo Hồ sơ Bệnh nhân (Create Patient Profile)

📝 Step-by-Step (Luồng thành công)

Step Tầng Hành động File
1 View Staff nhập form: Họ tên, Ngày sinh, SĐT, Email → nhấn "Tạo hồ sơ" create_patient_screen.dart
2 ViewModel createPatient() validate input (fullName, dob bắt buộc; email format) create_patient_viewmodel.dart
3 ViewModel Gọi _patientRepo.createPatientAndAccount(PatientEntity(...)) Dòng 51
4 Repository Tạo Medical Code: PHR-{DOB_clean}-{today}-{seq} → VD: PHR-15081990-25032026-01 patient_repository_impl.dart
5 Repository Mở db.transaction() Dòng 32
6 DB (txn) Nếu có email/SĐT → INSERT user_accounts (role=CUSTOMER, password=SHA256('123456')) Dòng 34-59
7 DB (txn) Xác định family_id (= userId nếu có account, hoặc family_id của staff) Dòng 64
8 DB (txn) INSERT patient_profiles (medical_code, full_name, dob, phone, email, family_id) Dòng 66-78
9 DB (txn) Nếu có userId → INSERT family_access (relationship='Bản thân') Dòng 80-87
10 DB (txn) INSERT audit_logs (action='Tạo hồ sơ bệnh nhân') Dòng 90-97
11 Repository Commit transaction → Nếu có email → gửi Welcome Email (fire-and-forget) Dòng 100-101
12 ViewModel Set _isSuccess = true, _successMedicalCode = newCode → notifyListeners() Dòng 62-63
13 View Hiển thị dialog thành công với Medical Code, nút "Tạo mã truy cập" UI Screen

🔀 Luồng thay thế đã triển khai

ALT-1: Email/SĐT đã tồn tại (Unique Constraint Violation)

  • Step 6 → DatabaseException.isUniqueConstraintError() → kiểm tra email conflict
  • Nếu email đã có patient → throw 'Email này đã được đăng ký hồ sơ. Mã Y Tế: {code}'
  • Nếu email đã có account khác → throw 'Email này đã được sử dụng cho một tài khoản khác.'
  • ViewModel catch → set _errorMsg → View hiển thị lỗi

ALT-2: Bệnh nhân không có Email/SĐT (vd: trẻ em)

  • Skip tạo user_accounts (userId = null)
  • family_id lấy từ _getUserFamilyId(txn, createdByStaffId) (family của staff tạo)
  • Không gửi welcome email

ALT-3: Validation lỗi ở ViewModel

  • fullName hoặc dob rỗng → return false với _errorMsg
  • Email không chứa '@' → return false

2.2. Luồng: Liên kết Thành viên Gia đình (Family Linking)

📝 Step-by-Step (Luồng thành công)

Step Tầng Hành động
1 View Customer nhập: Medical Code + Access Code + Quan hệ (Vợ/Chồng, Con, Bố/Mẹ, Anh/Chị/Em)
2 ViewModel linkMember() gọi _patientRepo.linkFamilyMember(...)
3 Repository Validate medical_code + access_code trong patient_profiles
4 DB (txn) Tự chữa lành: Nếu linker chưa có 'Bản thân' → tạo patient_profile + family_access cho linker
5 DB (txn) INSERT family_access (customerId → targetPatientId, relationship)
6 DB (txn) INSERT system_notifications (thông báo liên kết thành công)
7 DB (txn) Family ID Sync: Cập nhật family_id cho target profile + target account nếu head
8 DB (txn) Liên kết đối xứng: Nếu target có account → tạo reverse link (VD: Con↔Bố/Mẹ)
9 DB (txn) Khám phá mạng lưới: Scan existing family_access → auto-link nếu logic khớp
10 DB (txn) Cross-sharing: Chia sẻ danh sách hồ sơ con giữa A và B
11 ViewModel Refresh familyMembers list → notifyListeners()

🔀 Luồng ngoại lệ đã triển khai

  • Mã y tế/mã truy cập sai → throw 'Mã y tế hoặc mã truy cập không chính xác.'
  • Linker chưa có hồ sơ 'Bản thân' → tự tạo patient profile + liên kết (self-healing)
  • family_id null → tự patch family_id = customerId

2.3. Luồng: Đăng nhập (Login)

Step Tầng Hành động
1 View User nhập email/SĐT + password, chọn loại (Customer/Staff)
2 ViewModel login() validate input → gọi _authRepo.login(email, password, isCustomer)
3 Repository Hash password SHA-256 → query user_accounts với role filter + status='ACTIVE'
4 Repository Nếu CUSTOMER → kiểm tra patient_profile có bị LOCKED không
5 ViewModel Nếu rememberMe → lưu session vào SharedPreferences
6 View Navigate → CustomerFamilyHome / AdminDashboard / StaffDashboard

🔀 Luồng ngoại lệ

  • Sai credentials → 'Thông tin đăng nhập không chính xác hoặc sai loại tài khoản.'
  • Profile bị LOCKED → throw PatientProfileLockedException → 'Hồ sơ đã bị khóa'
  • Session restore (main.dart) → Nếu userId đã lưu nhưng user bị xóa → clear session → force re-login

2.4. Luồng: Tạo/Quản lý Tài liệu Y tế (Medical Document CRUD)

📝 Step-by-Step (Tạo tài liệu)

Step Tầng Hành động
1 View Staff chọn bệnh nhân → nhập: category, title, notes, record_date, files, tags
2 ViewModel Gọi _docRepo.createDocument(doc)
3 Repository INSERT medical_documents → Ghi audit_logs
4 Repository Nếu status='SAVED' → gửi notification cho customer liên quan
5 Repository Loop addFileToDocument() → Copy file vật lý vào app directory + INSERT document_files
6 Repository Loop addTagToDocument() → Upsert tag (case-insensitive) + INSERT document_tags

Soft Delete → Restore → Hard Delete

  • Soft Delete: is_deleted=1, status='DELETED', lưu status_before_soft_delete
  • Restore: is_deleted=0, khôi phục status gốc
  • Hard Delete: Xóa file vật lý → xóa document_files → xóa document_tags → xóa medical_documents
  • Clear Trash: Batch hard delete tất cả docs đã soft-delete của 1 staff

2.5. Luồng: Tạo Đơn Khám Bệnh (Medical Exam)

Step Tầng Hành động
1 View Staff chọn bệnh nhân → nhập: ngày khám, chẩn đoán, triệu chứng, dấu hiệu sinh tồn, đơn thuốc, ghi chú, ngày tái khám
2 ViewModel Gọi _examRepo.createMedicalExam(exam)
3 Repository Lấy category_id của "Đơn Khám Bệnh" (tạo nếu chưa có)
4 Repository Map entity → MedicalExamModel.toDocumentJson(categoryId) → INSERT medical_documents (tái sử dụng bảng documents)
📝 Ghi chú thiết kế Medical Exam tái sử dụng bảng medical_documents thay vì bảng riêng. Thông tin exam (symptoms, diagnosis, prescription, vital_signs) được serialize vào field notes và title.
